# nanobot

基于 `HKUDS/nanobot` 深度改造的个人分支，现作为独立开发主线维护。

这个分支已经不再是对上游的轻量同步版本。  
它的核心运行时、记忆系统、上下文组装方式和长期行为治理边界都已经发生明显分叉，更偏向一个**可治理的个人助理型 agent**，而不是单纯沿用原始项目叙事。

## 项目定位

本分支关注的是：在保持整体 agent 骨架可运行、可恢复、可扩展的前提下，把短期状态、长期记忆、归档材料、候选观察明确分层，并用结构化对象替代“原始消息尾部即当前状态”的老式做法。

当前版本重点强调：

- 运行时短期状态由结构化状态对象驱动
- `working/CURRENT.md` 只作为镜像 / handoff 视图
- 历史归档与当前工作内存彻底分离
- 长期身份层不再允许被 Dream 直接改写
- 长期行为改变必须经过候选层与晋升闸门

## 为什么做这次分叉

原始 agent 项目在轻量场景下很好用，但一旦进入长期使用、持续交互、自动总结、后台心跳、自我反思这类更复杂的个人助理场景，几个问题会越来越明显：

- 短期状态和历史材料容易混在一起
- 一次错误总结可能直接污染长期行为
- `CURRENT.md` 这类文本摘要既承担镜像，又承担“主真相源”，职责过重
- 恢复、提交、压缩、Dream 消费的对象不统一
- 后台自动流程如果没有权限边界，容易把未经审核的结论直接写成“人格事实”

因此，这个分支的核心方向不是“让 agent 记得更多”，而是：

**让不同类型的信息待在该待的位置，并限制哪些信息有资格真正影响未来行为。**

## 当前架构概览

这个分支当前采用的是一套**分层、分权限、结构化状态优先**的架构。

### 1. 运行时短期状态

当前短期运行时主链位于 `.nanobot/state/`，以结构化对象为主，而不是依赖自由文本摘要。

关键对象包括：

- `TurnState`：当前这一轮在做什么、做到哪一步
- `WorkingSetSnapshot`：当前稳定工作集
- `TurnCapsule`：一轮完成后的结构化结论
- `ArtifactRecord`：工具结果及其可复用投影
- `CommitManifest`：一轮是否“真正稳定完成”的正式判据

其中：

- `WorkingSetSnapshot` 是当前稳定短期工作内存
- `CommitManifest` 是回合完成与恢复的正式边界
- `TurnCapsule` 和 `ArtifactRecord` 负责把“可复用结论”和“可复用证据”沉淀下来

## 2. 上下文组装方式

当前 prompt 不再简单地把整段历史消息直接塞给模型，而是按固定结构组装：

1. `System Prefix`
2. `WorkingSetSnapshot`
3. `Recent Raw Turns`
4. `Selected TurnCapsules`
5. `Selected Artifact Render`
6. `Current User Message`

也就是说：

- 工作集优先于原始历史
- 原始历史只保留最近合法尾部
- 大工具结果默认以 digest / render 视图进入 prompt
- `working/CURRENT.md` 即使出现，也只是镜像层，不再是主状态真相源

## 3. 记忆分层

当前分支把记忆明确拆成四层：

### `identity/`
高权限长期记忆层，包含稳定行为原则、稳定用户规则、稳定用户画像。  
这层默认直接影响未来行为和系统 prompt。

### `working/`
当前工作镜像层。  
`working/CURRENT.md` 仍然存在，但它的职责是：

- 给人读
- 做 handoff
- 做兼容文本层
- 作为 Dream 可更新的镜像输出

它不再承担运行时主状态和恢复判定职责。

### `archive/`
归档与反思层。  
保存历史材料、反思结果、检索底库，但不再承担当前主工作内存职责。

### `candidate/`
候选观察层。  
用来暂存尚未正式升权的观察、推断和晋升建议，避免摘要结果直接污染高权限身份层。

## 4. 长期记忆如何进入身份层

当前版本最重要的治理边界之一，是长期行为更新链路：

`committed turn -> Dream -> candidate observations -> Promoter -> identity/*`

这条链路的含义是：

- `Dream` 可以总结、提炼、归档、提出候选
- `Dream` 不能直接改写 `identity/*`
- `Promoter` 才是正式的升权闸门
- 只有通过候选层审核的结论，才有资格进入高权限长期记忆

这个设计的核心目标不是“摘要永远正确”，而是：

**即便摘要有误，也不能轻易获得过高权限。**

## 5. 与上游 nanobot 的主要差异

本仓库最初 fork 自 `HKUDS/nanobot`，仍继承部分原始代码和整体 agent 骨架。  
但当前实现已经在以下方面与上游显著分叉：

- 记忆架构从统一仓库式叙事改为分层分权
- 短期上下文从“历史尾部驱动”切到“结构化工作集驱动”
- `CURRENT.md` 从主状态降级为镜像 / handoff 层
- 回合完成不再只靠消息链推断，而是以 `CommitManifest` 为正式边界
- 长期身份层引入 `candidate -> Promoter -> identity` 的独立晋升链

因此，如果你阅读这个仓库，请优先以本仓库的文档与代码为准，不要默认沿用上游对 memory、context、Dream 的旧理解。

## 6. 适合谁

这个分支更适合以下方向的探索：

- 长期运行的个人助理型 agent
- 需要 heartbeat / dream / cron 类后台流程的 agent
- 重视记忆权限边界和长期行为治理的 agent
- 希望把运行时状态、归档、候选结论、身份层分开的场景
- 需要更稳定的 handoff、恢复、提交语义的本地 agent
