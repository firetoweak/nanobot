# Architecture Documentation

本目录用于维护 `local_dev` 分支当前真实实现的项目架构文档。

目标有两个：

- 让新接手的人能在较短时间内理解项目的核心能力、边界和关键约束。
- 让 AI 在修改代码前后，都能快速定位受影响的能力域，并同步更新对应文档。

关于 `docs/` 目录：

- `docs/` 视为历史设计记录与持续更新的草稿区。
- 其中内容可能过期、未落地，或只代表某一阶段的分析，不应默认视为当前实现。
- 后续做设计判断、改代码或更新架构文档时，不应以 `docs/` 作为主依据。
- 当前有效依据的优先级应为：代码真实实现 > `Architecture_Documentation/` > 其他说明性文档。

本目录采用以下组织方式：

- `overview/`：项目总览、核心原则、全局运行流和长期约束。
- `capabilities/`：按能力/职责域组织的正式架构文档，是主阅读入口。
- `mapping/`：源码目录、测试目录与能力域文档的映射，以及“改哪些代码要改哪些文档”的规则。
- `adr/`：关键设计取舍记录。仅在出现重要架构决策或边界调整时新增或修改。
- `changes/`：实验性设计、未定型方案、近期差量变更记录。验证稳定后再提升到正式文档。

阅读顺序建议：

1. `overview/project-overview.md`
2. `overview/design-principles.md`
3. `capabilities/` 下与当前任务最相关的文档
4. `mapping/code-to-capability.md`
5. 如涉及设计取舍，再看 `adr/`

真实性约束：

- 文档必须描述“当前真实实现 + 明确设计意图”。
- 不得把尚未实现、仅部分实现或仍处实验状态的内容写成既成事实。
- 如实现与设计意图暂不一致，必须显式写出差异和现状。

维护规则摘要：

- 默认采用差量更新，只更新本次改动直接影响的 capability 文档和 mapping。
- 当职责边界、核心流程、长期约束变化时，再更新 `overview/`。
- 当出现重要设计取舍时，再新增或修改 `adr/`。
- 实验性内容先进入 `changes/`，稳定后再提升。
