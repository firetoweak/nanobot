# Documentation Maintenance Workflow

本文件说明每次 AI coding 或人工改动后，如何最小成本维护架构文档。

## 1. 默认流程

1. 先定位本次改动影响的代码路径。
2. 在 `mapping/code-to-capability.md` 中找到对应能力域文档。
3. 只更新直接受影响的 capability 文档。
4. 如边界变化，再决定是否更新 `overview/`。
5. 如出现关键取舍，再决定是否写入 `adr/`。
6. 如实现尚未稳定，先写到 `changes/`。

## 2. 每类文档的更新触发条件

### 2.1 `capabilities/`

以下情况默认要更新：

- 外部行为变化
- 内部职责边界变化
- 关键限制条件变化
- 真实实现与原文档不再一致

### 2.2 `overview/`

仅在以下情况更新：

- 项目主定位变化
- 核心运行流变化
- 长期设计原则变化

### 2.3 `adr/`

仅在以下情况更新：

- 正式采纳了新的关键设计取舍
- 旧取舍被替代
- 需要保留设计决策背景，便于后续不走回头路

### 2.4 `changes/`

优先承接以下内容：

- 实验性实现
- 仍在验证的方案
- 尚不能写成正式能力边界的短期变更

## 3. 更新时必须写清楚的内容

- 当前真实实现是什么
- 明确的设计意图是什么
- 哪些能力已经稳定
- 哪些能力是兼容层、过渡层或部分实现

## 4. 禁止写法

- 把未完成功能写成已完整实现
- 用“应该”“未来会”替代当前事实，却不说明现状
- 只写理想结构，不写当前兼容层和技术债
