# 设计原则

本文件描述当前 `local_dev` 分支维护时应遵守的长期原则。它们不是“所有地方都已完全做到”的宣传语，而是解释当前代码为什么朝这个方向设计。

## 1. 真实实现优先于理想化描述

- 架构文档必须优先反映当前真实实现。
- 如果设计意图领先于实现，必须明确标注“意图”与“现状”的差距。
- 兼容层、过渡层、遗留路径必须写明，不能默认它们已经消失。
- `docs/` 下的历史分析或草稿不能直接当作当前事实引用，必须回到代码和正式架构文档核对。

## 2. 能力域优先于源码目录

- 文档主索引按能力/职责域组织，不按源码目录组织。
- 源码目录和测试目录只在 `mapping/` 中维护映射。
- 这样做是为了让读者先理解系统职责，再回到具体实现文件。

## 3. 结构化状态优先于自由文本镜像

当前项目的重要方向，是把运行时真相逐步从自由文本摘要迁移到结构化状态对象。

当前体现包括：

- `.nanobot/state` 下有独立的 turn、message、response、working-set、capsule、artifact、commit 存储。
- `working/CURRENT.md` 仍存在，但作为镜像/兼容层，不再承担唯一真相源职责。
- prompt 组装优先消费 working set、capsule 和 artifact，而不是无限追加原始历史。

## 4. 长期记忆需要治理，不允许任意写高权限层

当前设计明确区分：

- `identity/`：高权限长期记忆。
- `working/`：当前工作镜像。
- `archive/`：归档与反思。
- `candidate/`：候选观察与待晋升结论。

原则上，自动总结结果不应直接写入高权限身份层；必须经过候选层和晋升规则。

## 5. 多入口共享同一执行内核

- CLI、channels、HTTP API、SDK 不应各自拥有分叉的 agent 内核语义。
- 当前统一收敛到 `AgentLoop`。
- 各入口允许有输入协议差异，但不应各自复制一套上下文、工具、状态和模型调用逻辑。

## 6. 差量维护优先

文档维护策略不是“每次全量重写”，而是：

- 默认只更新本次改动直接影响的 capability 文档和 mapping。
- 只有职责边界、核心流程或长期约束变更时，才更新 `overview/`。
- 只有关键设计取舍出现或变化时，才更新 `adr/`。
- 未稳定方案先进入 `changes/`。

## 7. 测试是架构事实的一部分

- 对本项目来说，测试目录不仅是质量保障，也是当前行为边界的重要证据。
- 当代码行为主要通过测试锁定时，能力文档应明确引用相关测试域。
- 文档不应写出与测试事实相矛盾的断言。
