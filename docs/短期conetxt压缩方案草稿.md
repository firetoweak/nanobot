# 短期上下文压缩方案草稿（历史文档）

## 1. 文档状态

本文保留为历史草稿，用来记录 nanobot 从旧短期上下文方案迁移到结构化状态主链之前的设计背景与问题意识。

它不是当前实现的权威说明。

如果你要了解现行实现，请优先阅读：

- `docs/上下文工作集优化方案.md`
- `docs/回合状态恢复优化方案.md`
- `docs/MEMORY.md`

## 2. 这份草稿还保留什么价值

本文仍然有价值的部分包括：

- 为什么旧 `session.messages` 尾部驱动会导致上下文膨胀
- 为什么单纯截断 tool message 会损失可复用证据
- 为什么恢复不能只做消息修补
- 为什么短期上下文需要显式工作集，而不是“最近消息越多越好”

这些问题陈述仍然成立，也解释了为什么项目后续走向结构化状态驱动。

## 3. 本文不再代表现状的部分

下面这些旧叙述已经不再代表当前主实现：

- 把 `Session.get_history()` 或 `session.messages[last_consolidated:]` 视为短期主输入
- 把 `working/CURRENT.md` 视为运行时主工作记忆
- 把 `archive/history.jsonl` 视为 Dream 推断当前短期状态的主入口
- 把 idle summary 注回看成短期连续性的核心机制
- 把 tool message 压缩视为主证据保留策略

这些路径在当前实现中已经降级为：

- 原始日志
- 兼容文本镜像
- 归档与检索输入
- 模型侧预算治理辅助

## 4. 旧方案与现行方案的差异

### 4.1 旧方案的重心

旧草稿的重心是：

- 压缩原始历史
- 控制 token 压力
- 给 Dream 和长期记忆提供归档材料

### 4.2 现行方案的重心

现行方案的重心已经变成：

- 用 `WorkingSetSnapshot` 表达当前稳定工作内存
- 用 `TurnState` 表达执行进度和恢复入口
- 用 `ArtifactRecord` 和 `TurnCapsule` 表达可复用证据与结论
- 用 `CommitManifest` 定义稳定完成判据

也就是说，当前重点不再是“如何更聪明地裁原始消息”，而是“如何让短期状态本身变成可恢复、可消费、可沉淀的结构化对象”。

## 5. 建议阅读方式

如果你是为了理解历史演进，可以把这份文档当成：

- 旧问题列表
- 设计动机记录
- 迁移前方案草稿

如果你是为了写代码、更新实现或解释当前架构，请不要以本文为准，而应以现行权威文档为准。
