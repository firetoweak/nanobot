# 回合状态恢复实现合同

## 1. 文档角色

本文是 nanobot 当前恢复链路的权威说明，描述 active turn 如何落盘、恢复、修复半提交，以及系统如何判定一个 turn 已稳定完成。

本文只描述当前实现，不再使用“准备迁移到”“最终会替换”之类的规划语气。

## 2. 当前恢复主链

当前恢复控制面围绕以下对象运行：

1. `TurnState`
2. `CommitManifest`
3. `MessageObject / ResponseObject`
4. `ArtifactRecord / TurnCapsule / WorkingSetSnapshot`
5. `active-turn.json / latest-turn.json / latest-working-set.json`

恢复入口优先读 `TurnState`，而不是重放原始消息历史。

## 3. 核心对象

### 3.1 TurnState

`TurnState` 是 active turn 的执行态真相源。

```python
TurnState = {
    "turn_id": str,
    "session_key": str,
    "task_ref": str | None,
    "execution_ref": str | None,
    "revision": int,
    "current_stage": str,  # collecting_user | awaiting_model | awaiting_tools | finalizing_turn | completed | interrupted
    "resume_action": str | None,  # await_tools | request_model_again | tail_finalize | replan
    "user_message_ref": str | None,
    "declared_tool_calls": list[dict],
    "completed_tool_results": list[dict],
    "artifact_refs": list[str],
    "capsule_ref": str | None,
    "capsule_status": str,
    "working_set_version": int | None,
    "injected_messages": list[dict],
    "injection_revision": int,
    "final_response_ref": str | None,
    "commit_id": str | None,
    "commit_state": str,  # open | sealing | committed | repair_needed
    "commit_manifest_ref": str | None,
    "error_state": dict | None,
    "created_at": str,
    "updated_at": str,
}
```

当前约束：

- `revision` 每次状态推进单调递增。
- `injection_revision` 在 mid-turn injection 时单调递增。
- `resume_action` 由当前状态重算并回写。
- `commit_state != committed` 时，turn 不能视为稳定完成。

### 3.2 CommitManifest

`CommitManifest` 是稳定提交对象，不是附属注释对象。

```python
CommitManifest = {
    "commit_id": str,
    "turn_id": str,
    "session_key": str,
    "turn_revision": int,
    "artifact_refs": list[str],
    "capsule_ref": str | None,
    "working_set_version": int | None,
    "final_response_ref": str | None,
    "completed_marker": bool,
    "created_at": str,
}
```

当前约束：

- manifest 必须存在且 `completed_marker=True`
- manifest 的 `turn_revision` 必须与稳定 turn revision 对齐
- manifest 中引用的 capsule / response / working set / artifact 必须可解析

### 3.3 统一引用协议

对象引用统一使用：

- `message:<message_id>`
- `response:<response_id>`
- `artifact:<artifact_id>`
- `capsule:<capsule_id>`
- `commit:<commit_id>`

业务层不应直接拼接路径跳过 `resolve_ref()`。

## 4. 状态机

### 4.1 合法阶段

```text
collecting_user
awaiting_model
awaiting_tools
finalizing_turn
completed
interrupted
```

### 4.2 合法迁移

```text
collecting_user -> awaiting_model
awaiting_model -> awaiting_tools
awaiting_model -> finalizing_turn
awaiting_tools -> finalizing_turn
finalizing_turn -> completed
any_active_stage -> interrupted
interrupted -> awaiting_tools | awaiting_model | finalizing_turn | completed
```

### 4.3 Injection 规则

当前实现把 mid-turn injection 视为当前 active turn 上的 revision bump，而不是新建 turn。

要求：

- 注入消息进入 `TurnState.injected_messages`
- `revision` 与 `injection_revision` 一起推进
- `resume_action` 会根据新状态重算
- 旧 revision 的工具结果不得被错误归入新的稳定提交

## 5. 唯一稳定完成判据

一个 turn 是否稳定完成，当前只看结构化提交链，而不是只看阶段名。

最小判定条件：

1. `TurnState.commit_state == committed`
2. `TurnState.commit_manifest_ref` 存在
3. `resolve_ref(commit_manifest_ref)` 得到有效 manifest
4. `is_completed_manifest(manifest)` 为真

恢复链路在此基础上还会继续验证：

5. manifest 的 `turn_revision` 与 `TurnState.revision` 一致
6. manifest 引用的 capsule / response / working set / artifact 都能加载

因此：

- `current_stage == completed` 不是唯一完成判据
- 只有 manifest 自洽且可解析时，completed 才成立

## 6. 当前恢复判定顺序

恢复 active session 时，当前逻辑按以下顺序工作：

1. 读取 `active-turn.json`
2. 加载对应 `TurnState`
3. 如果 `current_stage == completed` 或已有 `commit_manifest_ref`，先校验提交对象
4. 若提交对象不自洽，执行 `repair_partial_commit()`
5. 若修复后 manifest 自洽且 turn 已 completed，则：
   - 清空 `active-turn.json`
   - 推进 `latest-turn.json`
   - 不再把该 turn 当成未完成任务恢复
6. 若 turn 仍未完成，则把 `TurnState` 投影回合法消息序列，继续执行

这里的重点是：恢复优先依据结构化状态判断“该继续、该收口还是该修复”，而不是先从原始消息猜测阶段。

## 7. 当前发布协议

`AgentLoop.finalize_turn()` 当前按以下顺序写入稳定对象：

1. 归并 `artifact_refs`
2. 生成并写入 `TurnCapsule`
3. 生成并写入 `WorkingSetSnapshot`
4. 生成并写入 `ResponseObject`
5. 生成并写入 `CommitManifest`
6. 以 CAS 方式推进 `TurnState`：
   - `commit_id`
   - `commit_manifest_ref`
   - `final_response_ref`
   - `working_set_version`
   - `capsule_ref`
   - `commit_state=committed`
   - `current_stage=completed`
7. 推进 `latest-turn.json`
8. 仅当 working set 稳定时，推进 `latest-working-set.json`

这条顺序的含义是：

- 先落普通对象
- 再落 manifest
- 最后让 `TurnState` 指向稳定提交

## 8. repair_partial_commit() 合同

当前 `repair_partial_commit()` 的目标不是“猜一个完成结果”，而是保守地判定能否重新建立自洽 manifest。

处理步骤：

1. 先把 turn 推进到 `finalizing_turn + repair_needed`
2. 归并当前 artifact 引用
3. 检查以下对象是否齐全：
   - artifacts
   - capsule
   - final response
   - working set
4. 如果核心对象缺失，则把 turn 降级为：
   - `current_stage=interrupted`
   - `commit_state=repair_needed`
   - 清空提交相关 refs
5. 如果对象齐全，则重建 manifest 并回写：
   - `commit_manifest_ref`
   - `commit_state=committed`
   - `current_stage=completed`
6. 必要时补推进 `latest-turn.json` 和 `latest-working-set.json`

因此，repair 的结果只有两种：

- 修回 committed
- 保守降级为 interrupted，等待后续 `tail_finalize` 或 `replan`

## 9. Resume Action 合同

当前恢复动作至少包括：

- `await_tools`：仍有工具结果未齐，继续等待或补齐
- `request_model_again`：工具结果齐了，但没有稳定最终答复
- `tail_finalize`：只差收口与持久化
- `replan`：现有中间状态已不足以安全继续

`compute_resume_action()` 依赖当前 `TurnState` 判断下一步动作，而不是依赖旧 checkpoint 文本猜测。

## 10. 旧路径现状

### 10.1 已退出恢复主链

下列旧路径不再是恢复主入口：

- `runtime_checkpoint`
- `pending_user_turn`
- 仅根据 `session.messages` 修补 assistant/tool 链来判断阶段
- 仅根据 `_last_summary` 或 `CURRENT.md` 推断当前执行位置

### 10.2 仍保留但已降级

以下内容仍存在，但职责已经降级：

- `session.messages`：原始审计日志与兼容会话存储
- `last_consolidated`：原始会话归档边界
- `working/CURRENT.md`：镜像/交接文本
- `archive/history.jsonl`：归档与检索材料

它们都不是恢复控制面的主判据。

## 11. 索引文件职责

当前恢复控制面依赖的索引文件如下：

- `active-turn.json`：当前活动 turn
- `latest-turn.json`：最近一个稳定完成的 turn
- `latest-working-set.json`：最近一个稳定工作集
- `dream-cursor.json`：Dream 已处理的 committed turn 进度

这些索引共同组成当前结构化恢复和后续消费链路。

## 12. 对外结论

如果要回答“nanobot 现在怎么恢复一轮任务”，答案是：

- 执行态看 `TurnState`
- 稳定完成看 `CommitManifest`
- 提交修复走 `repair_partial_commit()`
- 最新稳定对象通过 `active-turn` / `latest-turn` / `latest-working-set` 索引定位

不要再把旧 checkpoint 或原始消息尾部当成恢复主真相源。
