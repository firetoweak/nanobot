# 回合状态恢复优化方案

## 1. 目标

本方案用于定义 nanobot 新恢复链路的实现合同。目标不是继续增强旧消息修补逻辑，而是让系统在中断后能够恢复当前 turn 的执行状态、稳定产物和下一步动作。

本方案的最终目标：

- 用 `TurnState` 替代 `runtime_checkpoint` / `pending_user_turn` 的恢复核心
- 用 `CommitManifest` 定义“稳定完成 turn”的唯一判据
- 用 `repair_partial_commit()` 处理半提交状态，而不是猜测 turn 已完成
- 让恢复、Prompt、AutoCompact、Dream 共用同一套稳定短期对象

## 2. 当前问题

当前恢复链路已经能修补消息一致性，但还不能可靠恢复执行状态。

### 2.1 当前恢复主要修消息，不修阶段

旧方案主要能处理：

- user message 提前落盘
- assistant/tool 断链
- 部分 tool result 缺失

但系统仍缺少显式机制去回答：

- 当前 turn 到了哪个阶段
- 下一步应该继续等工具、重新问模型还是只做尾部收口
- 哪些结果已经稳定，不该再重做

### 2.2 已完成结果与待完成步骤边界不清

当前缺少统一状态对象来明确表达：

- 哪些 tool call 已声明
- 哪些 tool result 已完成
- 哪些结果已持久化为 artifact
- 是否已生成 capsule
- 是否已刷新 working set
- 是否已生成最终响应

### 2.3 旧恢复链容易退化成重复执行

如果没有显式状态对象，恢复逻辑会重新读历史、重新推断、重新跑工具，导致：

- token 浪费
- 长任务稳定性下降
- 中断后的行为不可预测

## 3. 设计原则

- 恢复以状态为中心：优先恢复 `TurnState`，而不是重放旧消息历史
- 结果复用优先：已稳定落盘的 artifact / capsule / response / working set 默认复用
- 回合级建模：恢复粒度是 turn，而不是零散 message
- 状态与事实分层：原始日志保真，状态对象只表达执行位置和稳定产物
- 先可恢复，再删除旧链：迁移期允许双写，但最终必须删除旧恢复主链
- 最终不保留旧恢复架构：旧 metadata 只能是迁移窗口的一次性输入

## 4. 核心对象

本方案的恢复主链由以下对象构成：

1. `TurnState`
2. `CommitManifest`
3. `MessageObject / ResponseObject`
4. `ArtifactRecord / TurnCapsule / WorkingSetSnapshot`

### 4.1 TurnState

`TurnState` 是执行态唯一真相源。

建议字段：

```python
TurnState = {
    "turn_id": str,
    "session_key": str,
    "task_ref": str | None,
    "execution_ref": str | None,
    "revision": int,
    "current_stage": str,  # collecting_user | awaiting_model | awaiting_tools | finalizing_turn | completed | interrupted
    "resume_action": str | None,  # await_tools | request_model_again | tail_finalize | replan
    "user_message_ref": str | None,
    "declared_tool_calls": list[dict],
    "completed_tool_results": list[dict],
    "artifact_refs": list[str],
    "capsule_ref": str | None,
    "capsule_status": str,  # not_started | ready | persisted
    "working_set_version": int | None,
    "injected_messages": list[dict],
    "injection_revision": int,
    "final_response_ref": str | None,
    "commit_id": str | None,
    "commit_state": str,  # open | sealing | committed | repair_needed
    "commit_manifest_ref": str | None,
    "error_state": dict | None,
    "created_at": str,
    "updated_at": str,
}
```

约束：

- `revision` 每次写入必须单调递增
- `resume_action` 可以重算，但必须可回写
- `commit_state=committed` 之前，不得把 turn 当作稳定完成
- `*_ref` 一律使用统一对象引用协议

### 4.2 CommitManifest

`CommitManifest` 是稳定完成判据，而不是附属对象。

```python
CommitManifest = {
    "commit_id": str,
    "turn_id": str,
    "session_key": str,
    "turn_revision": int,
    "artifact_refs": list[str],
    "capsule_ref": str | None,
    "working_set_version": int | None,
    "final_response_ref": str | None,
    "completed_marker": bool,
    "created_at": str,
}
```

约束：

- 恢复、Dream、AutoCompact 只信任 manifest 是否有效
- `current_stage == completed` 不是唯一完成判据
- 若 manifest 缺失或不完整，必须进入 `repair_needed`

### 4.3 统一引用对象

```python
MessageObject = {
    "message_id": str,
    "session_key": str,
    "turn_id": str | None,
    "role": str,
    "content": Any,
    "created_at": str,
}

ResponseObject = {
    "response_id": str,
    "session_key": str,
    "turn_id": str,
    "source_revision": int,
    "content": Any,
    "created_at": str,
}
```

引用规范：

- `message:<message_id>`
- `response:<response_id>`
- `artifact:<artifact_id>`
- `capsule:<capsule_id>`
- `commit:<commit_id>`

## 5. Turn State Machine

### 5.1 合法阶段

```text
collecting_user
awaiting_model
awaiting_tools
finalizing_turn
completed
interrupted
```

### 5.2 合法迁移

```text
collecting_user -> awaiting_model
awaiting_model -> awaiting_tools
awaiting_model -> finalizing_turn
awaiting_tools -> finalizing_turn
finalizing_turn -> completed
any_active_stage -> interrupted
interrupted -> awaiting_tools | awaiting_model | finalizing_turn | completed
```

### 5.3 mid-turn injection 规则

injection 不是新 turn，而是当前 active turn 的 revision bump。

要求：

- 每次 injection 都必须推进 `revision` 和 `injection_revision`
- injection 默认允许更新 `last_user_focus`
- 若 injection 改变任务目标或约束，必须重算 `resume_action`
- 若旧工具计划前提被破坏，`resume_action` 必须转为 `request_model_again` 或 `replan`

## 6. Resume Action 合同

恢复后至少需要支持以下动作：

### 6.1 `await_tools`

适用条件：

- `current_stage == awaiting_tools`
- 存在未完成 `declared_tool_calls`
- 已有结果不足以收口

### 6.2 `request_model_again`

适用条件：

- 工具结果已齐
- 无稳定最终回复
- working set 已可组装

### 6.3 `tail_finalize`

适用条件：

- artifact 和工具结果已齐
- capsule / working set / final response 尚未全部完成
- 不需要再跑工具或重问模型

### 6.4 `replan`

适用条件：

- 状态缺关键字段
- artifact / capsule / tool result 校验失败
- working set 不可信
- injection 改变了旧计划前提

## 7. 恢复前一致性检查

恢复前至少校验：

- `declared_tool_calls` 与 `completed_tool_results` 的 `tool_call_id` 对齐
- `artifact_refs` 均可加载
- `working_set_version` 可加载到有效 snapshot
- `capsule_ref` 可加载且版本合法
- 若 `commit_state == committed`，则 manifest 必须存在且自洽

任一关键项失败时，不允许静默续跑，必须进入 `repair_needed` 或 `replan`。

## 8. publish / repair 协议

### 8.1 收口写入顺序

每个 turn 收口必须按如下顺序：

1. 写 artifact 原文或对象原文
2. 写 `ArtifactRecord`
3. 更新 `TurnState.completed_tool_results` 和 `artifact_refs`
4. 写 `TurnCapsule`
5. 写 `WorkingSetSnapshot`
6. 写 `ResponseObject`
7. 写 `CommitManifest(completed_marker=True)`
8. 用 `expected_revision` CAS 更新 `TurnState`，同时写入：
   - `commit_id`
   - `commit_manifest_ref`
   - `final_response_ref`
   - `working_set_version`
   - `capsule_ref`
   - `commit_state=committed`
   - `current_stage=completed`

要求：

- 不允许先把 `commit_state` 单独写成 `committed`
- 不允许在 manifest 缺失时把 turn 当作稳定完成
- 稳定消费方只看 manifest 是否存在且自洽

### 8.2 repair 规则

恢复时只要满足以下任一条件，就不能视为稳定完成：

- `commit_manifest_ref` 缺失
- manifest 缺失
- manifest 引用对象缺失
- `commit_state != committed`
- `turn_revision` 与 manifest 不一致

此时必须：

1. 将 `commit_state` 改为 `repair_needed`
2. 执行 `repair_partial_commit()`
3. repair 成功后才允许重新提交为 committed
4. repair 无法自洽时进入 `interrupted + replan`

## 9. revision 与 stale result 规则

### 9.1 revision 规则

- `TurnState.revision`：每次状态写入单调递增
- `TurnState.injection_revision`：每次中途注入单调递增
- 工具声明和结果必须带 `declared_revision`

### 9.2 stale result 规则

只有 `declared_revision == active_revision` 的结果才允许进入当前 stable commit。

旧 revision 的结果：

- 可以保留为审计 artifact
- 默认不得参与当前 turn 的 stable commit
- 不得继续驱动 `resume_action`

## 10. 与现有模块的整合

### 10.1 `nanobot/agent/loop.py`

重点改造：

- 创建和推进 `TurnState`
- 维护 active turn index
- 恢复时优先读取 `TurnState`
- 收口时执行 `finalize_turn()` / `repair_partial_commit()`

### 10.2 `nanobot/agent/runner.py`

重点改造：

- 记录 declared tool calls
- 上报 completed tool results
- 工具结果必须带 revision 信息
- 不得直接写 `TurnState`

### 10.3 `nanobot/session/manager.py`

重点改造：

- 提供 turn / message / response / commit / working set 的状态读写能力
- 提供 `resolve_ref()`
- 提供 `expected_revision` CAS 写入能力

### 10.4 `nanobot/agent/autocompact.py`

重点改造：

- 只消费 committed turn
- 不得通过旧自然语言 summary 推断当前阶段

### 10.5 `nanobot/agent/memory.py`

重点改造：

- Dream 只消费稳定完成 turn
- 不得继续通过 `CURRENT.md + history.jsonl` 反推当前短期执行状态

## 11. 旧恢复链迁移原则

迁移窗口内允许旧 metadata 作为一次性输入，但不允许继续做主恢复源。

### 11.1 迁移矩阵

| 旧状态 | 新状态目标 | 处理规则 |
| --- | --- | --- |
| 仅有 `pending_user_turn` | `collecting_user` 或 `interrupted` | 指向最后一条 user message，保守设为 `tail_finalize` 或 `replan` |
| 仅有 `runtime_checkpoint` | `awaiting_model` / `awaiting_tools` / `finalizing_turn` | 根据 checkpoint 中 assistant/tool 信息映射 |
| 两者并存 | `interrupted` | 优先保留 user 触发，再映射 checkpoint，不得猜成 completed |
| `_last_summary` 空闲态 | snapshot 候选元数据 | 不得直接注回 prompt |
| 只有原始日志 | 无 active turn | 不臆造 turn 完成 |

### 11.2 最终态约束

迁移窗口结束后，以下旧主路径必须删除：

- `runtime_checkpoint`
- `pending_user_turn`
- 依赖消息修补猜状态的恢复主链

## 12. 分阶段实施要求

### 阶段 0

先同步文档中的恢复合同，不改生产代码。

### 阶段 1

先落地状态目录和 schema，不切主恢复链。

### 阶段 3

切换 `AgentLoop` 到 `TurnState` 驱动，并建立迁移 helper。

### 阶段 5

切换恢复和稳定消费方到 manifest / repair 协议。

### 阶段 7

删除旧恢复主路径并完成端到端回归。

## 13. 验收标准

本方案完成后，必须满足：

- 恢复优先读取 `TurnState`
- 已完成 turn 的判断只看 `CommitManifest`
- 半提交状态能进入 `repair_partial_commit()`
- stale revision 的工具结果不会被错误提交
- 旧 `runtime_checkpoint` / `pending_user_turn` 不再是恢复主入口

## 14. 总结

本方案的核心不是继续增强“消息修补”，而是把恢复链路重构为：

- 一个显式的 turn 状态机
- 一套稳定对象和提交清单
- 一条可 repair、可复用、可保守回退的恢复路径

最终目标是让系统在中断后稳定回答三件事：

- 这轮做到哪一步了
- 哪些结果已经稳定完成
- 下一步应该继续执行、重新规划，还是只做收口
