# 设计总结

## 文档定位

- 这是一份轻量设计偏好文档，不是开发事实记录。
- 只有在做方案判断、跨模块改动、重构关键链路或确认长期红线时，才优先快速参考这里。
- 如果与当前用户需求或当前代码冲突，以当前需求和当前代码为准。
- 详细实现过程、历史改动证据看 `SUMMARISE.md`。
- 每次代码改动后的设计取舍和经验看 `设计总结日记.md`。
- 这里不要机械更新，只保留当前仍然有效的偏好、红线和阶段重点。

## 与 `.cursor/rules/` 的配合方式

- 是否需要先看这份文档，由 `.cursor/rules/design-context.mdc` 决定；这里负责提供被命中后应快速继承的内容。
- 是否需要更新这份文档，由 `.cursor/rules/summary.mdc` 的“长期偏好变化 / 新红线 / 新轻量规则 / 阶段重点变化”来触发。
- 这份文档只给方向，不替代当前代码和当前需求，也不替代 `SUMMARISE.md` 的事实记录职责。

## 适合记录什么

- 当前仍然有效、未来还会反复用到的设计偏好。
- 已经验证过的长期红线，避免后续重复踩坑。
- 当前阶段最值得持续投入的治理重点。
- 少量模块级特殊约束，只保留真正长期有效的部分。

## 不适合记录什么

- 某一次改动的事实过程、验证步骤和文件清单。
- 只影响单次任务的临时取舍。
- 还没有被验证、容易过时的想法或待讨论方案。
- 为了“保持同步”而复制 `SUMMARISE.md` 或 `设计总结日记.md` 的内容。

## 🎯 通用工程哲学与 AI 约束（可跨项目复用）

- [架构] 复杂链路优先保证可观测性，必要时使用独立业务 trace，而不是只依赖接口摘要日志。
- [逻辑] 输入治理优先在结构化字段层处理，避免在最终 prompt 字符串层做硬截断。
- [修复] 优先做单点修复，避免为了修一个局部问题就重构整条链路。
- [判断] 设计建议要尽量继承已验证有效的方向，但不能压过当前需求和当前代码现状。
- [红线] 不要在最终 prompt 字符串末尾硬截断来解决输入过大问题。

## `nanobot` 项目特有红线与偏好（仅限本项目有效）

- [记忆分层] 常驻 prompt 保持“小而稳”，只注入 `identity/*` 和 `working/CURRENT.md`；`archive/*` 与 `candidate/*` 默认走检索，不进入常驻区。
- [权限分级] `identity` 不接受 Dream 或 heartbeat 直写；身份升权必须经过独立审查闸门，而不是在归档或反思阶段顺手落盘。
- [默认出口] Dream 和 heartbeat 的默认写出口应落在 `working`、`archive`、`candidate`，不要把普通经历直接提升成长期人格或稳定用户规则。
- [兼容迁移] 结构升级优先采用“新结构主写、旧结构兼容读”的渐进迁移，避免一次性硬切路径把现有 workspace、命令和历史数据打断。
- [工具权限] 后台/系统来源的 agent 要按来源分级授权，尤其是 heartbeat 这类自动执行链路，不应复用普通对话的全量写权限。
- [职责边界] `Dream` 负责观察、归档和候选提炼，`Promoter` 负责升权；不要把观察、归档、晋升重新揉回一个组件里。
- [排障] 记忆问题优先检查“写入层级是否正确、是否被常驻注入、是否越权写入”，不要只看某个 markdown 文件有没有被改。
- [维护] 不要为了“保持同步”机械更新这份文档。

## 📅 当前阶段重点（随时间推移可删除或更新）

- 观察 `Dream -> candidate -> Promoter -> identity` 这条新链路的真实产物质量，确认候选提炼和晋升规则是否足够稳。
- 继续清理旧 `SOUL.md / USER.md / memory/history.jsonl` 心智模型，把文档、命令和使用说明逐步迁到新分层结构。
- 根据真实运行反馈，逐步增强 `candidate/observations.jsonl` 的冲突处理、过期失效和修订策略。
- 保持 heartbeat、cron 等自动执行链路的权限边界清晰，避免后续功能叠加时重新把身份层写权限放开。

## 模块级特别说明

### Memory / Context

- `identity/*` 负责稳定人格、长期用户规则和长期用户画像；`working/CURRENT.md` 负责高频短期状态。
- `archive/*` 和 `candidate/*` 是检索底库，不应默认参与每轮 prompt 组装。
- 如果系统 prompt 又开始膨胀，先检查是否有人把 archive/candidate 重新塞回常驻上下文。

### Dream / Promoter

- Dream 默认应产出 archive note、candidate observation、promotion proposal，而不是直接改身份层。
- `Promoter` 的第一职责是“阻止错误升权”，第二职责才是“把稳定事实写进 identity”。
- 当需要加新晋升规则时，优先加在 `Promoter`，不要回退到 Dream 直接写 identity。

### Heartbeat / 自动执行链路

- heartbeat 之类的自动执行链路要用受限 agent 或等价权限边界，确保默认只能更新 `working` 或 `archive`。
- 如果未来要开放更多后台写能力，必须先明确路径级授权，而不是直接复用主 agent 工具集。

## 维护原则

- 新增内容前，先问一句：这条结论三次任务后还值得保留吗？
- 如果结论只服务于一次改动，优先写进 `设计总结日记.md`。
- 如果结论需要事实证据支撑，优先把事实写进 `SUMMARISE.md`，这里只保留抽象后的偏好或红线。
- 每次更新尽量是小幅增删，不把这里写成大而全手册。
