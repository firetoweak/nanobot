# 设计总结日记

## 文档定位

- 这里记录每次代码改动后的设计取舍、坑和经验。
- 这是增量日记，不是开发事实主记录；事实、实现和验证请看 `SUMMARISE.md`。
- 这里按次追加，不删除历史内容。
- 这里比 `设计总结.md` 更具体，但不承担长期轻量偏好的维护职责。

## 何时追加

- 命中“里程碑改动”时追加一条当次设计日记。
- 典型场景包括：跨模块改动、对外行为变化、重要 bugfix、明确方案取舍、沉淀出可复用经验。
- 重点写方案取舍、踩坑、经验，不重复抄实现细节。
- 如果某次结论已经上升为长期偏好，再考虑是否需要同步更新 `设计总结.md`。

## 通常不追加的场景

- 单文件局部修复，且没有明显设计取舍。
- 纯注释、文案、命名整理。
- 试验性排查或一次性辅助改动。
- 已经能在最终回复中说清楚、没有长期复用价值的小改动。

## 初始化记录

### 2026-04-10 初始化

- 核心问题：需要把“长期轻量偏好”和“每次改动后的设计经验”拆开，同时把两份文档初始化成可长期沿用的结构。
- 采用方案：`设计总结.md` 只保留长期有效的偏好、红线和阶段重点；`设计总结日记.md` 只记录里程碑改动中的设计取舍、限制和经验。
- 为什么这样选：这样既不会让长期文档越写越厚，也能在真正重要的改动后留下足够有用的设计上下文。
- 本次暴露的坑 / 限制：如果日记触发条件过宽，还是会退化成流水账；如果长期总结混入示例脚手架，后续 AI 容易把示例当现行规则。
- 对后续同类问题的提醒：先判断这次结论是“长期偏好”还是“单次经验”，再决定写到 `设计总结.md`、`设计总结日记.md` 还是只写进最终回复。

### 2026-04-17 权限分级记忆改造

- 核心问题：NanoBot 原来把 Dream 放得太靠近长期身份层，普通经历经 `history -> Dream` 提炼后可以直接改 `SOUL.md / USER.md / MEMORY.md`，而这些内容又会常驻进入上下文，一次错误抽象就可能变成后续行为偏置。
- 采用方案：没有重写整套 memory，而是在原骨架上把长期层拆成 `identity / working / archive / candidate` 四层；Context 只常驻 `identity + working`；Dream 默认只写 `working`、`archive`、`candidate`；再新增独立 `Promoter` 负责 `candidate -> identity` 的硬规则晋升；heartbeat 另外走受限 agent。
- 为什么这样选：这样能把变化集中在“升权边界”和“常驻注入边界”上，而不是重做 session、consolidation、history、Dream 的整条主链；同时用兼容读和迁移入口保住已有 workspace，不必一次性硬切旧路径。
- 本次暴露的坑 / 限制：即使做了分层，第一版 `Promoter` 规则仍然偏硬编码，`candidate` 的冲突、过期和修订策略还比较轻；另外旧 `memory/MEMORY.md`、根目录 `SOUL.md / USER.md` 还在兼容期，团队心智如果不及时更新，后续很容易又把旧路径当主路径使用。
- 对后续同类问题的提醒：凡是涉及“是否应该进入长期身份层”的问题，先问是不是应该先落 `candidate`；凡是涉及“是否应该常驻进 prompt”的问题，先问是不是只该检索不该注入；凡是后台自动任务需要写文件时，先定义路径级授权，不要直接复用主 agent 的全量工具权限。

### 2026-04-17 旧版生产文档迁移收口

- 核心问题：虽然代码已经支持新分层记忆，但真实工作目录里仍并存根目录 `SOUL.md / USER.md` 与 `memory/MEMORY.md`，而且旧副本和新 `identity/*` 内容已经分叉，继续兼容下去只会让后续人工维护更容易写错地方。
- 采用方案：先做迁移前快照备份，再把旧根路径文档中的有效内容合并进 `identity/SOUL.md`、`identity/USER_PROFILE.md`、`identity/USER_RULES.md`；确认 `archive/*` 已承担真实归档职责后，直接删除工作目录中的旧根路径文档与 `memory/MEMORY.md`，只保留新分层作为主生产地址。
- 为什么这样选：代码层已经有兼容读，继续保留旧文件的价值很低，反而会持续制造“双主路径”错觉；把兼容留在实现层、把主路径收口到文档层，能最快修正团队心智。
- 本次暴露的坑 / 限制：`USER.md` 这类旧文件往往同时混着“用户画像”和“行为规则”，迁移时不能机械复制，必须按语义拆到 `USER_PROFILE` 和 `USER_RULES`；另外 README 与独立 memory 文档如果不同时更新，工作区删干净了，仓库说明还是会把人带回旧路径。
- 对后续同类问题的提醒：凡是从“兼容期”走向“正式主路径”的改造，不要只做代码 fallback；还要同步清理真实工作区里的旧副本、更新说明文档，并把迁移结果写进版本日志，否则旧心智会比旧代码活得更久。
